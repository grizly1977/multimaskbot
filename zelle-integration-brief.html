// phase 4 — developer brief

Zelle & Gmail Integration

📅 3 травня 2026 v1.0 Status: Draft Bank: Chase Business

Система автоматично обробляє вхідні Zelle-платежі через парсинг email-сповіщень від Chase Bank. Кожен платіж співставляється з клієнтом через базу алиасів та зараховується на баланс як стандартний top-up.

⚑ Ключовий принцип

Для решти системи (SPARK, баланс, webhooks) Zelle top-up нічим не відрізняється від карткового top-up. SPARK не знає що це Zelle — для нього це просто поповнення балансу.

// 3.5 — email parsing

Email Parsing: Chase Zelle Notification

Система отримує HTML email від Chase і парсить з нього 6 полів. Нижче — реальний приклад email та точне пояснення що і звідки братиметься.

Крок 1 — Фільтр: чи це Zelle-платіж?

Перевірити Subject заголовок email. Якщо точний збіг — обробляти. Якщо ні — ігнорувати.

Subject header: You received money with Zelle®

⚠ Зверни увагу: Інші Chase emails (баланс, виписки, refund-повідомлення) мають інший Subject і будуть відфільтровані автоматично. Додатково перевірити що From містить @chase.com для безпеки.

// що і звідки парсити

1 sender_name DB field
Де: HTML тіло — тег <h1>
Raw: "HONEST REPAIR LLC sent you money"
Парсинг: Відрізати суфікс " sent you money" (завжди незмінний)
Результат: "HONEST REPAIR LLC"
2 amount_cents DB field
Де: HTML таблиця, рядок з лейблом "Amount", права колонка
Raw: "$10.00"
Парсинг: Видалити $, parseFloat, × 100, round → integer
Результат: 1000 (центи)
3 zelle_transaction_id DB field UNIQUE
Де: HTML таблиця, рядок "Transaction number", права колонка
Raw: "29078018871"
Парсинг: Взяти як є (string). Використовується для дедупликації та reconciliation з CSV
Результат: "29078018871"
4 email_received_at DB field
Де: Email header Date: (НЕ з поля «Sent on» в HTML — там немає часу)
Raw: "Sun, 3 May 2026 21:43:52 +0000 (UTC)"
Парсинг: Парсити як RFC 2822 datetime → UTC
5 email_message_id DB field
Де: Email header Message-ID:
Raw: <701751780.2806034.1777844632345.JavaMail.webuser@iaasn00692056>
Використання: Аудит + дедупликація Pub/Sub push retries (той самий лист = той самий Message-ID)
6 chase_notification_id DB field
Де: Email header NOTIFICATION-ID:
Raw: NOTIFICATION-84657_digital-pymt-comm-service_..._Zelle-4155-cdde943d-5ef9-3a30-a2f5-f27582abe6ec
Використання: Debug Pub/Sub retry-логіки. Зберігати як є (string)

Крок 3 — Нормалізація sender_name перед matching

Після парсингу з <h1> — виконати нормалізацію перед пошуком в ZelleAlias:

1. Відрізати " sent you money" суфікс з кінця рядка
2. .trim() — видалити пробіли з початку і кінця
3. .toLowerCase() — порівняння з alias завжди case-insensitive
// приклад
"HONEST REPAIR LLC sent you money"
→ strip suffix → "HONEST REPAIR LLC"
→ trim → "HONEST REPAIR LLC"
→ lowercase → "honest repair llc" ← порівнюється з alias DB

Технічні нюанси HTML парсингу

Quoted-Printable encoding

Email тіло закодовано у quoted-printable. Перед парсингом HTML необхідно декодувати:

encoded: =3D → =
encoded: =C2=AE → ®
line wrap: рядок що закінчується на = → об'єднати з наступним

Що ігнорувати

ignored Рядок "Sent on" — дата є в email header Date: з часом
ignored Рядок "Memo" — не використовується для matching взагалі
ignored Весь footer (disclaimer, посилання, "Go to Zelle" кнопка)
ignored Текст-підтвердження внизу: "HONEST REPAIR LLC is registered with a Zelle® member bank..."

// 6 — reconciliation

Reconciliation Flow (Daily/Weekly)

Щоденна звірка з банківським CSV. У майбутньому CSV замінюється на Plaid API. Конфігурується через ZELLE_RECONCILIATION_CRON (default: 0 8 * * * — 8:00 щодня).

flowchart TD
    A(["⏰ ZelleReconciliationJob\ncron trigger"]) --> B["Admin uploads CSV\n(або майбутній Plaid pull)"]
    B --> C["Parse CSV:\nextract Zelle incoming transactions"]
    C --> D["For each transaction in CSV:"]
    D --> E{{"zelle_tx_id\nє в ZelleTransaction?"}}

    E -->|ТАК — існує| F["Verify amounts match"]
    F -->|✓ OK| G(["✅ All good"])

    E -->|НІ — пропущена| I["CREATE ZelleTransaction\nsource = csv"]
    I --> J["Run alias matching\n(як нова транзакція)"]
    J -->|FOUND| K["Auto top-up"]
    J -->|NOT FOUND| L["Create ticket\nstatus = pending"]
